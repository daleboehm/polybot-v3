## 2. Executive Summary

### The system in one paragraph

Polybot-v2 is a TypeScript dual-engine Polymarket trader (Prod `polybot-v2.service` + R&D `polybot-v2-rd.service`) running on a single DigitalOcean VPS (178.62.225.235:2222). The surface is clean — strict TypeScript, zero `as any`, zero `@ts-ignore`, parameterised SQL, pino structured logs, graceful shutdown wiring, a sub-strategy refactor (schema v2) deployed the day of the audit — but the muscle behind that surface is missing or wrong in ways that together explain the prod-blocking symptom Dale flagged: **40 open positions, $2.49 cash, no new trades in days**. The system has been kept alive by Dale's full-time attention. **It will not survive an unattended weekend.**

### The single most important finding

**The Position Resolution Pipeline is structurally broken in at least four ways that compound**, and any one of them would produce the observed symptom:

1. Gamma API query uses non-standard plural `condition_ids` parameter that Gamma almost certainly ignores (`resolution-checker.ts:182`)
2. Response shape cast to `GammaMarket[]` without validating it isn't `{data: [...], next_cursor}` (`resolution-checker.ts:194`)
3. Three silent `return null` paths in `parseGammaResolution` with zero telemetry — catches JSON.parse errors, length mismatches, and missing >0.95 winner; logs only at `debug` level (`resolution-checker.ts:206-238`)
4. **A v1 Python cron (`auto_redeem.py`) is still running on the VPS, redeeming positions on-chain that v2 doesn't know about** — v2's DB shows positions open while the wallet has long since been swept. No reconciliation ever reconciles them.

Compounding this: position `current_price` is **never updated** after fill (`updatePositionPrice` is dead code with zero callers), so unrealized P&L is always 0 and the stop-loss monitor — which is **also a no-op, only logs, never executes sells** (`engine.ts:130-137`) — has nothing to monitor anyway. Cash drainage from a **hardcoded 2% phantom fee on every BUY fill** (Polymarket actually charges 0%) compounds from the other direction (`clob-router.ts:124-125`, `paper-simulator.ts:44-48`).

**Net effect**: positions go in, never come out, cash drains from phantom fees, advisor promotes strategies on trade volume that includes stuck positions, sizer multiplies by a strategy weight *after* the per-position cap (doubling it), strategies like `value`/`skew`/`complement` fabricate edge from arithmetic on the market price, the engine never reconciles against the exchange on startup, and nothing alerts when any of this goes wrong.

### 4.1 Position Resolution Pipeline

This is the prod-blocking section. If only one thing gets fixed, fix this.

**The symptom**: Prod has 40 open positions and $2.49 cash. Positions never close. Cash never returns. The engine cannot place new trades.

**The four compounding causes**:

| Cause | Evidence | Why it breaks resolution |
|---|---|---|
| **R-1: Gamma `condition_ids` plural parameter** | `resolution-checker.ts:182` | Polymarket Gamma uses singular `condition_id`. Repeated plural params are non-standard and likely ignored. If Gamma falls back to a default list, zero of the prod positions match, and every position records `skipped_not_found`. |
| **R-2: Response shape unverified** | `resolution-checker.ts:194` | Response cast as `GammaMarket[]` with zero validation. Gamma can return `{data: [], next_cursor: null}` for some endpoints. `for (const market of data)` either iterates empty or throws "object not iterable" (caught silently at the outer `try/catch`). |
| **R-3: `parseGammaResolution` three silent null paths** | `resolution-checker.ts:206-238`, described in full in C-P1-1 | `JSON.parse` failure → `catch { return null }` (no log). Length mismatch → silent. No winner >0.95 → silent (logs at `debug`, not surfaced). Markets in `resolution_pending` with stale 0.5/0.5 prices get skipped and never retried. |
| **R-4: v1 `auto_redeem.py` silently drains the wallet v2 thinks it owns** | `Polymarket/auto_redeem.py:138-143`, `:245-289` | v1 cron calls `NegRiskAdapter.redeemPositions(...)` on-chain. v2 has no knowledge. v2's `positions` table stays `open` for markets that v1 has already redeemed. v2's `entity.cash_balance` diverges from the actual on-chain wallet. |

**The additional compounding factors**:

- Position `current_price` is **never updated** after the fill (`position-repo.ts:61-70` — `updatePositionPrice` has zero callers across `src/`, confirmed via Grep). Unrealized P&L is permanently 0. Stop-loss and profit-target conditions can never fire.
- Stop-loss monitor is a **no-op** (`engine.ts:130-137`). The scan produces `ExitSignal[]`, the engine iterates them, the loop body just calls `log.info(...)`. There is no SELL order construction. The comment on line 133 literally says `// Process exits (would generate sell signals)`.
- Phantom 2% fee (`clob-router.ts:124-125`, `paper-simulator.ts:44-48`) drains cash on every BUY fill by $0.10-$0.40 per $5-$20 trade. Over 40 positions this is $4-$16 of unaccounted drainage — the entire current cash delta.

**The better solution (the user asked for one)**:

Replace the Gamma-polling mechanism with **Polymarket Data API `/positions?user={proxy_wallet}`** as the source of truth. This endpoint returns the wallet's actual on-chain position list, including a `redeemable` flag and realized P&L. On each scan cycle:

1. Query `https://data-api.polymarket.com/positions?user={proxy_wallet}` (one call, returns all positions, no per-market Gamma fan-out).
2. For each DB position in `getAllOpenPositions()`:
   - If present in the API response with `redeemable: false` → still open, do nothing.
   - If present with `redeemable: true` → call v1's `NegRiskAdapter.redeemPositions` inline (or rely on v1 cron and just record it), then mark DB position closed with the API's realized P&L.
   - If **absent** from the API response → it's already been redeemed (by v1 or by Polymarket). Close it in the DB. Back-compute the payout from the USDC wallet delta.
3. For any API position **not** in the DB → the engine placed an order the DB failed to record (a crash-window bug). Insert it with a reconstructed entry price.

This is the reconciliation R-6 demands. It makes resolution authoritative from the on-chain truth rather than from Gamma's cached metadata. It also lets v2 safely coexist with the v1 cron: both see the same source of truth.

**Secondary path**: directly query the Polygon RPC for the wallet's `balanceOf` for each position's CTF ERC-1155 token ID. Zero balance = redeemed. More expensive in RPC calls but no API dependency.

**Either path also requires**: deleting the phantom 2% fee, wiring the stop-loss exits through `routeOrder`, and calling `updatePositionPrice` every cycle so unrealized P&L is real.

## 6. Backtesting & R&D Validity

(Track E raw findings: `Polymarket/audit/2026-04-09/E-backtest-rnd-validity.md` — 17 findings. Full math in the raw file; summary here.)

### 6.1 Advisor thresholds are statistically meaningless at current defaults

`strategy-advisor.ts:148-172` defaults at `schema.ts:63-69`:

```
ENABLE  if total_resolutions ≥ 10 AND win_rate ≥ 50% AND total_pnl > 0
DISABLE if total_resolutions ≥ 20 AND win_rate < 30% AND total_pnl < 0
```

**Wilson 95% binomial confidence intervals** at these thresholds:

| Scenario | n | k | Wilson 95% CI | Can reject p=0.5? |
|---|---|---|---|---|
| **Enable at threshold** | 10 | 5 | **[0.237, 0.763]** | **No** |
| Enable "promising" | 10 | 6 | [0.313, 0.832] | No |
| "Clear edge" | 20 | 12 | [0.387, 0.781] | No |
| Borderline | 80 | 48 | [0.498, 0.692] | Borderline |
| **Statistically significant** | **100** | **60** | **[0.503, 0.689]** | **Yes** |
| **Disable threshold** | 20 | 5 | [0.112, 0.468] | We may be killing true 47% WR strategies |

**Interpretation**: a sub-strategy promoted at n=10, 50% WR could have a true win rate anywhere from **24% to 76%**. The advisor is not distinguishing signal from noise — it is amplifying noise. The disable threshold is symmetrically broken: we're pulling the plug on strategies that might be fine.

**Fix**: `min_resolutions_to_enable = 100` (where Wilson LB crosses 0.5 at 60% observed), `min_resolutions_to_disable = 100`, or switch to a sequential probability ratio test / Thompson sampling approach with a prior.

### 6.2 Zero walk-forward validation anywhere in the codebase

Grep confirms no `walk-forward`, `walk_forward`, `train_test_split`, `oos`, `out-of-sample`, `cross_validation`, or `holdout` references in `polybot-v2/src/`. The strategies are trained (parameter choices were made during development) on the same data they're evaluated on — a textbook in-sample overfit setup. The `v_strategy_performance` view is effectively training P&L, not validation P&L.

**Fix**: introduce expanding-window walk-forward at the sub-strategy level. `agi-walk-forward-validation` specifies the pattern.

### 6.3 13 of 15 sub-strategies have tautological probability models

Track E found that `value`, `skew`, `complement`, `convergence`, and their variants compute `model_prob` as arithmetic on `market_price` — e.g., `value.ts:64-65` sets `model_prob: price + 0.05`. A strategy whose model is "the market price plus a constant" mathematically cannot have edge against the market price. Kelly sizing on such a model is a random walk with drift toward ruin.

**The two strategies that do have real models**: `favorites` (price-range-based, defensible empirical thesis from the 8-strategy research list) and `longshot` (FADE logic, also defensible). These two should remain; the other 13 should either be rewritten with real data-driven models or removed.

### 6.4 No risk-adjusted metrics (Sharpe, Sortino, Calmar, Brier, reliability)

`v_strategy_performance` tracks: total_trades, total_resolutions, wins, losses, win_rate, total_pnl. Missing: Sharpe ratio, Sortino, max drawdown, Calmar, Brier score (calibration), reliability diagram data, distribution of returns, time-weighted return. Without these, the advisor cannot distinguish a strategy that won 60% of the time with high variance from a 55% strategy with low variance — the latter is better risk-adjusted but would lose the promotion race.

### 6.5 R&D ↔ live path divergence

Track E found four material divergences between the R&D paper path and the live path:

1. **Fee handling** — R&D applies the phantom 2% fee; live applies it too. Same bug, but the *symmetry* means R&D P&L is systematically understated in both directions.
2. **Position price updates** — A-P0-3 affects both; R&D unrealized is also permanently 0.
3. **Stop-loss monitor** — also no-op in R&D; R&D has no exits either.
4. **Strategy weighter** — live disables it (engine.ts:62-63), R&D uses it. The effective sizer is different, so R&D's "proven" strategies are sized differently from what they'd get in live.

Net effect: R&D is not simulating live execution; it's simulating "live with weighter" + same bugs. The promotion signal is based on a different execution profile than the prod engine will apply.

## 9. Finding Format Convention

Every finding in this report follows the same structure:

- **ID**: Track letter + severity + sequence (A-P0-1, B-P0-3, etc.)
- **Severity**: P0 (immediate financial risk), P1 (high-priority correctness), P2 (correctness improvements), P3 (cleanup/hardening)
- **Evidence**: `file:line` with enough context to locate it without reading surrounding code
- **Root cause**: Per `obra-root-cause-tracing` — the *why*, not just the *what*
- **Impact**: What breaks, in user-observable terms
- **Recommended fix**: Concrete, implementable, not abstract
- **Effort**: S (< 1h), M (1-4h), L (1+ days)
- **REBUILD scope?**: Yes/No — whether this belongs in the rebuild or is surgical
- **Skills cited**: Which new skill(s) framed the finding

### Rec 1 — Replace resolution-checker with on-chain reconciliation (unblocks prod)

*Skill: `polymarket-official-agent`, `polymarket-trading-expert`, `obra-defense-in-depth`*

Create `src/market/data-api-reconciler.ts`:

```typescript
import { log } from '../core/logger.js';
import type { Entity } from '../types.js';

interface DataApiPosition {
  conditionId: string;
  asset: string;        // token_id
  size: number;         // shares
  avgPrice: number;
  currentValue: number;
  realizedPnl: number;
  redeemable: boolean;
  endDate: string;
}

export class DataApiReconciler {
  constructor(private baseUrl = 'https://data-api.polymarket.com') {}

  async getWalletPositions(proxyWallet: string): Promise<DataApiPosition[]> {
    const url = `${this.baseUrl}/positions?user=${proxyWallet}`;
    const res = await fetch(url, { signal: AbortSignal.timeout(10_000) });
    if (!res.ok) throw new Error(`Data API ${res.status}: ${await res.text()}`);
    return await res.json() as DataApiPosition[];
  }

  async reconcile(entity: Entity, dbPositions: Position[]): Promise<ReconcileResult> {
    const apiPositions = await this.getWalletPositions(entity.proxy_wallet);
    const apiByTokenId = new Map(apiPositions.map(p => [p.asset, p]));

    const toClose: Array<{pos: Position, reason: string, payoutUsd: number}> = [];
    const toInsert: DataApiPosition[] = [];

    for (const dbPos of dbPositions) {
      const api = apiByTokenId.get(dbPos.token_id);
      if (!api) {
        // Absent from API → already redeemed. Back-compute payout from cost basis history.
        toClose.push({ pos: dbPos, reason: 'redeemed_off_chain', payoutUsd: 0 /* reconcile from wallet delta */ });
      } else if (api.redeemable) {
        // Resolved on-chain, waiting for redemption. Record the realized P&L.
        toClose.push({ pos: dbPos, reason: 'resolved_redeemable', payoutUsd: api.currentValue });
      } else {
        // Still open — update current_price from API's realized+unrealized
        // (skip here, let upsertPosition handle the price refresh separately)
      }
      apiByTokenId.delete(dbPos.token_id);
    }

    // Any API position not matched is a DB orphan (crash window)
    for (const api of apiByTokenId.values()) {
      toInsert.push(api);
    }

    return { toClose, toInsert };
  }
}
```

Wire into `engine.ts` `runScanCycle`:

```typescript

  try {
    const reconciler = new DataApiReconciler();
    const dbPositions = this.positionRepo.getOpenPositions(entity.slug);
    const result = await reconciler.reconcile(entity, dbPositions);
    for (const { pos, reason, payoutUsd } of result.toClose) {
      await this.closePosition(pos, reason, payoutUsd);
    }
    for (const orphan of result.toInsert) {
      log.warn({ entity: entity.slug, orphan }, 'Orphan position found via reconciliation');
      // Insert reconstructed position row
    }
  } catch (err) {
    log.error({ entity: entity.slug, err }, 'Reconciliation failed — skipping scan cycle');
    continue; // fail closed: don't trade if reconciliation failed
  }
}
```

```typescript
class KillSwitch {
  private halted = false;
  private reason = '';

  halt(reason: string): void {
    this.halted = true;
    this.reason = reason;
    log.warn({ reason }, 'KILL SWITCH ACTIVATED');
  }

  isHalted(): boolean { return this.halted; }
}
```

```typescript
function wilsonLowerBound(wins: number, n: number, z = 1.96): number {
  if (n === 0) return 0;
  const phat = wins / n;
  const denom = 1 + z*z/n;
  const center = (phat + z*z/(2*n)) / denom;
  const halfWidth = z * Math.sqrt(phat*(1-phat)/n + z*z/(4*n*n)) / denom;
  return center - halfWidth;
}

// strategy-advisor.ts
if (perf.total_resolutions >= 30 && wilsonLowerBound(perf.wins, perf.total_resolutions) > 0.5 && perf.total_pnl > 0) {
```
